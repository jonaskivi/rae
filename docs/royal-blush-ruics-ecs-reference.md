# Royal Blush RUICS and ECS reference

This document describes the Royal Blush UI system in enough detail that another agent could recreate a similar system in Rae.

The system is called RUICS in the codebase. It is a scene-driven UI runtime built on top of:

- a lightweight ECS world
- JSON-authored scene files with the extension `.raescene`
- a small set of layout and rendering systems
- optional runtime-only behavior systems layered on top of the authored scene graph

The design goal is not "HTML-like layout". It is a deterministic, inspectable, game-UI scene graph where authored component data is the main source of truth.

## 1. High-level architecture

At a high level RUICS consists of five layers:

1. ECS storage layer
2. Scene serialization layer
3. Layout and transform systems
4. Render synchronization backend
5. Higher-level runtime behaviors such as scene instancing, data-bound lists, image source resolution, text binding, masking, wobble, smoke, and animation

The important architectural principle is:

- Authored `.raescene` files describe nodes and persisted components.
- Runtime systems read those components and derive measured, computed, and rendered state.
- Some runtime-only subtrees may be mounted dynamically, but authored layout still remains the base model.

## 2. ECS data model

### 2.1 Entity ids

Entities are integer ids.

- `EntityId = number`
- New ids are generated by incrementing `nextEntityId`.
- Alive entities are tracked in a `Set<EntityId>`.

### 2.2 Component storage

Each component type is stored in its own dense sparse table.

The table implementation stores:

- `denseEntities: EntityId[]`
- `denseData: T[]`
- `sparseIndex: Map<EntityId, number>`

Operations:

- `has(id)`
- `get(id)`
- `add(id, data)`
- `remove(id)`
- `entries()`
- `ids()`
- `data()`

This is a classic sparse-set style ECS storage, but implemented with arrays plus a map instead of a fixed sparse array.

### 2.3 UiWorld

`UiWorld` is the central ECS world.

It contains:

- entity allocation and liveness
- one `ComponentTable<T>` per component type
- a `revision` counter that increments when the world changes
- helper functions such as `query(...)`, `markChanged()`, `getAliveEntities()`, and destroy/clear operations

Key methods:

- `createEntity()`
- `ensureEntity(id)`
- `destroyEntity(id)`
- `clearEntity(id)`
- `isAlive(id)`
- `getAliveEntities()`
- `getRevision()`
- `markChanged()`
- `query(...tables)`

`query` is implemented by choosing the smallest component table and filtering ids that exist in all requested tables.

### 3.3 Component type details

Below is the high-level vocabulary and semantics.

#### Geometry and layout

##### `Rect`

Base authored rectangle:

- `x`
- `y`
- `w`
- `h`

This is the seed rectangle. It is not always the final rendered rectangle.

##### `Size`

Per-axis sizing policy:

- `w: SizeAxis`
- `h: SizeAxis`

Where `SizeAxis` is:

- `mode: 'Fixed' | 'Hug' | 'Fill'`
- optional `min`
- optional `max`

Semantics:

- `Fixed`: use the authored rect size
- `Hug`: use measured content size if available, otherwise fallback to authored rect size
- `Fill`: use the parent-provided available size if available, otherwise fallback to authored rect size

##### `Layout`

Container layout mode:

- `type: 'None' | 'Horizontal' | 'Vertical' | 'Grid' | 'Stack'`
- optional `gap`
- optional `alignMain`
- optional `alignCross`
- optional `columns`
- optional `rowGap`
- optional `columnGap`

Important note: `Grid` exists in the type, but the provided `LayoutSystem` implementation primarily handles `Horizontal`, `Vertical`, `Stack`, and `None`.

##### `Padding`

Four-sided inner padding:

- `l`
- `t`
- `r`
- `b`

##### `Margin`

Declared in the type system, but not meaningfully used by the layout algorithm shown here.

##### `SafeArea`

Safe-area-driven padding augmentation:

- `enabled: boolean`
- `apply: { l, t, r, b }`
- optional `extra`

`SafeAreaSystem` populates `SafeInsets` per entity, and `LayoutSystem` merges them into effective padding.

##### `Align`

Per-node alignment hints:

- `x?: 'Start' | 'Center' | 'End'`
- `y?: 'Start' | 'Center' | 'End'`

Used differently depending on context:

- In `Stack` layout, align positions the child inside the stack container.
- In row/column layout, align feeds into cross-axis alignment.
- When parent layout is `None`, align is not what positions the node; direct rect and offset do.

##### `Offset`

Explicit positional adjustment:

- `x`
- `y`

Applied after layout placement.

##### `Constraints`

Optional min/max clamps:

- `minW`
- `maxW`
- `minH`
- `maxH`

##### `OverflowPolicy`

Current modes:

- `none`
- `clip`
- `scaleToFitY`

In current code, `scaleToFitY` is implemented for vertical layout containers. It computes a `layoutScale` when required content height exceeds available height.

#### Visual and interaction

##### `TransformFx`

Visual transform properties:

- `scaleX`
- `scaleY`
- `rotation`
- `visible`
- `alpha`
- optional `pivot`
- optional `anchor`

This is not the layout rectangle. It is the post-layout visual transform metadata used by `TransformSystem` and `RenderSyncSystem`.

##### `Sprite`

- `textureKey`
- optional `tint`
- optional `scaleMode: 'Fit' | 'Fill' | 'Stretch'`
- optional `tileScale`
- optional `nineSlice` edges

A sprite entity can render as:

- normal sprite
- nine-slice sprite
- tiling sprite

depending on component data.

##### `Text`

- `text`
- optional `styleId`
- optional `wrapWidthMode: 'NodeWidth'`

`wrapWidthMode: 'NodeWidth'` is important. It makes wrapping use the node's computed width instead of a hardcoded style width.

##### `Shape`

- `kind: 'Rect' | 'RoundedRect' | 'Circle'`
- optional `fill`
- optional `stroke`
- optional `strokeWidth`
- optional `radius`

##### `Shadow`

Currently only:

- `blur`

This is backend-driven eye candy, not layout.

##### `Opacity`

- `value: number`

This multiplies effective alpha in `TransformSystem`.

##### `PointerEvents`

- `enabled: boolean`
- optional `cursor`
- optional `blockChildren`

##### `HitArea`

- `kind: 'Rect' | 'RoundedRect'`
- optional `radius`

##### `OnClick`

- optional `actionId`
- optional `actionIdDouble`
- optional `actionIdTriple`
- optional `maxDelayMs`

This component does not itself dispatch behavior. It is action metadata.

##### `Button`

- optional `role`

Semantic marker, used for tooling and authoring intent.

##### `Active`

- `value: boolean`

This is a runtime visibility/participation switch used throughout scenes for stateful variations.

##### `EditorVisible`

- `value: boolean`

##### `EditorLocked`

- `value: boolean`

Used by tooling/editor behavior.

#### Identity and authoring metadata

##### `Name`

Human-readable label.

##### `PrimaryType`

Semantic node type string such as `Button` or `Text`.

##### `NodeId`

Stable authored identifier string, used heavily in serialization and overrides.

##### `SceneScope`

Runtime field used to scope nested scene instance nodes so `sourceNodeId` and overrides can prefer the nearest scene instance subtree.

##### `RuntimeOnly`

Marks dynamically created runtime nodes that should not participate in normal authored parent layout flow.

#### Dynamic scene and data binding

##### `SceneInstance`

- `sceneId`
- optional `params`

This is one of the most important components. It means:

- "load another `.raescene` file into this entity"
- preserve this entity as the mounted root
- load the referenced scene under it
- then apply params, variants, and overrides

##### `ImageSourceResolver`

- `mode: 'Fixed' | 'Random' | 'Hash' | 'Direct'`
- `textureKeys: string[]`
- optional `seed`
- optional `once`
- optional `sharedHash`

This lets authored data describe texture selection policies without hardcoding the texture at author time.

##### `TextBinding`

- `key`
- optional `format: 'Plain' | 'Number' | 'Percent1' | 'Coins' | 'HandRank'`
- optional `prefix`
- optional `suffix`

This connects a text node to a runtime value resolver.

##### `MaskShape`

- `kind: 'Circle' | 'RoundedRect'`
- optional `sourceNodeId`
- optional `radius`

Used by `MaskShapeSystem`.

##### `DataRequest`

- `endpoint`
- `mode: 'Top' | 'AroundSelf'`
- optional `initialLimit`
- optional `aroundBefore`
- optional `aroundAfter`
- optional `pageLimit`
- optional `accountIdSource: 'None' | 'ActiveAccountId'`
- optional `itemsField`
- optional `cursorBeforeField`
- optional `cursorAfterField`

This is an authored declaration for paged server data.

##### `ListView`

- `itemSceneId`
- `itemKeyField`
- `itemHeight`
- optional `itemGap`
- optional `visibleItemCount`
- optional `overscanRows`
- `bindings: ListViewBinding[]`
- optional `loadingSceneId`
- optional `emptySceneId`
- optional `errorSceneId`

`ListViewBinding` is:

- `nodeId`
- `component: 'Text' | 'ImageSourceResolver' | 'Active'`
- `field`
- `itemField`
- optional `prefix`
- optional `suffix`

This lets one item scene be reused and rebound to many records.

#### Effects and motion

##### `BackgroundPan`

Scrolling/parallax background metadata:

- `speedX`
- `speedY`
- `amplitudeX`
- `amplitudeY`
- optional tab-parallax settings

##### `Carousel`

- `axis`
- optional `swipeThreshold`
- optional `transitionDuration`
- optional `travelFactor`
- optional `previewScale`

##### `SmokeFx`

Particle-style smoke configuration with spawn interval, jitter, lifetime, drift, scaling, and fade parameters.

##### `AnimFrames`

Frame sequence declaration:

- optional `baseKey`
- optional `frames: number[]`
- optional `textureKeys: string[]`
- `fps`
- optional `loop`
- optional `onEndTextureKey`

##### `AnimTrigger`

- `event: 'OnTabSelected' | 'OnAction' | 'Manual'`
- optional `actionId`
- optional `restartOnTrigger`

##### `AnimState`

Runtime animation state:

- `playing`
- `elapsed`
- `frameIndex`

##### `WobbleFx`

Procedural sine-wave motion settings.

## 4. Scene file format

### 4.1 File type

RUICS scene files are JSON files with extension `.raescene`.

Validated file format:

```json
{
  "type": "Scene",
  "version": 2,
  "sceneId": "profile",
  "root": "ProfileRoot",
  "nodes": {
    "ProfileRoot": {
      "Rect": { "x": 0, "y": 0, "w": 1080, "h": 2280 },
      "Size": { "w": { "mode": "Fill" }, "h": { "mode": "Fill" } },
      "Layout": { "type": "Stack" },
      "Children": ["ChildA", "ChildB"]
    }
  }
}
```

Validation rules:

- `type` must be `Scene`
- `version` must be `2`
- `sceneId` must be a string
- `root` must be a string and must exist in `nodes`
- every node id must match the internal node id pattern
- every child reference in `Children` must point to an existing node

### 4.2 Nodes are a dictionary, not an array

Nodes are stored in a map:

- key = stable `nodeId`
- value = `NodeV2`

This is extremely important:

- child order is handled by `Children`
- node identity is stable
- tooling can patch one node by id
- nested scene overrides can target nodes by id

### 4.3 Children references

Hierarchy is expressed by:

- `Children: [childNodeId1, childNodeId2, ...]`

Children are not nested inline as objects.

### 4.4 Optional components in nodes

A node can carry any combination of the supported persisted components.

There is no mandatory inheritance or widget class hierarchy. A node becomes meaningful through the component combination.

Examples:

- `Sprite` only: image node
- `Text` only: text node
- `Shape` only: vector panel
- `Layout + Children`: container
- `SceneInstance`: nested reusable scene
- `DataRequest + ListView`: virtualized server-backed list host

## 5. Scene loading and scene references

### 5.1 Layout key to file map

RUICS has an explicit scene id to file path map in code.

Examples:

- `main-menu` -> `assets/layout/main-menu.raescene`
- `shop` -> `assets/layout/shop.raescene`
- `deck` -> `assets/layout/deck.raescene`
- `ranking` -> `assets/layout/ranking.raescene`
- `profile-avatar` -> `assets/layout/profile-avatar.raescene`
- `ranking-profile-item` -> `assets/layout/ranking-profile-item.raescene`
- `chest-slot` -> `assets/layout/chest-slot.raescene`

This means `.raescene` files do not directly contain a path. They refer to another scene by logical `sceneId`, and code resolves that id to a file path.

### 5.2 Deserialization model

`deserializeLayoutV2`:

- validates the JSON
- creates or reuses ECS entities
- writes component data into the corresponding component tables
- reconstructs hierarchy using `HierarchySystem`
- installs defaults such as `TransformFx` if needed
- returns a map from fully scoped node id to entity id

A fully scoped node id looks like:

- `sceneId/nodeId`

Example:

- `profile/ProfileAvatarInstance`

### 5.3 Root reuse

A scene can be loaded into an existing root entity.

This is crucial for:

- editor reloads
- nested scene instances
- preserving parent placement while swapping content

`prepareLayoutRootForLoad`:

- collects subtree ids below the chosen root
- clears those entities
- preserves the chosen root entity
- returns a `reuseMap` of existing child node ids

Then `deserializeLayoutV2(..., { rootEntityId, reuseMap })` repopulates the subtree.

## 6. SceneInstance mechanics

This is one of the most important parts of RUICS.

### 6.1 What a SceneInstance means

A node with:

```json
"SceneInstance": {
  "sceneId": "profile-avatar",
  "params": { ... }
}
```

means:

- fetch the `profile-avatar` scene template
- use the current entity as the mounted root
- replace or populate its subtree with the template nodes
- keep the mounting entity's authored placement and root identity
- then apply params

### 6.2 Placement preservation

Before loading a nested scene instance, the system captures the mounting entity's:

- `Rect`
- `Size`
- `Align`
- `Offset`
- `Constraints`
- `OverflowPolicy`
- `TransformFx`

After loading the template, these root placement values are restored.

This is why a `SceneInstance` behaves like a reusable prefab placed into a local slot rather than like a full scene reset.

### 6.3 Scene scope

Each loaded scene instance subtree gets a `SceneScope` string.

This is used so operations such as:

- `MaskShape.sourceNodeId`
- `SceneInstance` overrides

can prefer matching nodes within the current instance subtree before falling back globally.

### 6.4 Variants

There is support for variant-style params.

Example from `profile-avatar`:

- `variant: 's'`

Runtime code can interpret variant strings and adjust specific nodes in the instance.

In Royal Blush this is used to scale and offset the avatar art for different avatar sizes.

### 6.5 Overrides

The most powerful scene-instance feature is generic overrides.

Example:

```json
"params": {
  "overrides": [
    {
      "nodeId": "ProfileAvatarRoot",
      "component": "OnClick",
      "field": "actionId",
      "value": "profile.avatar.open"
    }
  ]
}
```

Override semantics:

- target a node by `nodeId` inside the mounted subtree
- choose a component by name
- optionally choose a field inside that component
- set or remove the value

If `field` is omitted, the whole component object may be replaced or removed.

This is how reusable scenes become context-specific without duplicating scene files.

### 6.6 Nested scene instances

Scene instances may contain other scene instances.

Resolution order:

1. Repeatedly resolve unresolved scene instances until nothing new appears.
2. Run a second param/override pass across all instances.

That second pass matters because a parent instance may override nodes created by a child instance.

## 7. Layout semantics in detail

This is the heart of the system.

### 7.1 Two-phase layout

`LayoutSystem.update()` does two passes:

1. `measureSubtree`
2. `computeSubtree`

The first pass computes `MeasuredSize` from children.
The second pass computes actual `ComputedRect` placement and sizes.

### 7.2 Layout roots

Layout roots are entities with `Rect` and either:

- no parent
- or a special runtime-only boundary where a runtime-only subtree is mounted under an authored parent

This lets runtime-only subtrees have their own layout pass without affecting authored parent flow.

### 7.3 Active filtering

Children used by layout are filtered by runtime activity.

Inactive children do not contribute to measurement or placement.

This matters for:

- switching variations with `Active`
- hiding list status scenes
- modal visibility

### 7.4 `resolveAxis`

Axis resolution is simple and deterministic:

- if no `Size` or mode `Fixed`: use `Rect.w` or `Rect.h`
- if mode `Fill`: use parent-provided content size, else fallback to rect value
- if mode `Hug`: use measured size, else fallback to rect value

This means authored rect size is both:

- the fixed-size value for `Fixed`
- the fallback seed for `Fill` and `Hug`

### 7.5 Measurement rules

Measurement uses child resolved sizes and parent layout type.

#### Horizontal

- width = sum of child widths + gaps
- height = max child height

#### Vertical

- height = sum of child heights + gaps
- width = max child width

#### Stack

- width = max child width
- height = max child height

#### None

In the current implementation, `Layout: None` still measures as max child extents, not true arbitrary positioned extents.

Then parent padding is added and constraints are applied.

### 7.6 Padding and safe area

Effective padding = authored `Padding` plus optional safe area inset contribution.

Safe area behavior is per-edge.

### 7.7 Absolute rect computation when parent layout is None

If a node's parent has no layout, then:

- child authored `Rect.x/y` is effectively local to the parent's padded content origin
- parent padding is added
- then local `Offset` and `RuntimeOffset` are added

This is the direct-placement mode.

### 7.8 Container-driven layout when parent has Horizontal or Vertical layout

When the parent has layout:

- parent content box is computed from parent `ComputedRect` minus effective padding
- each child gets size resolved against content width/height
- fill children share remaining main-axis space
- `alignMain` controls initial cursor or gap distribution
- `alignCross` plus per-node `Align` control cross-axis position
- `Offset` and `RuntimeOffset` are applied after layout placement

### 7.9 `Horizontal` layout semantics

Main axis = width.
Cross axis = height.

The child main size is width. Cross size is height.

`alignMain` options:

- `Start`
- `Center`
- `End`
- `SpaceBetween`

`alignCross` options:

- `Start`
- `Center`
- `End`
- `Stretch`

### 7.10 `Vertical` layout semantics

Main axis = height.
Cross axis = width.

Same alignment semantics as horizontal, but rotated conceptually.

### 7.11 `Stack` layout semantics

All children occupy the same content box.

Each child is sized independently, then aligned inside the box using its own `Align`.

Default if `Align` missing:

- center on both axes

This makes `Stack` the RUICS equivalent of an overlay layout.

It is used for:

- modal overlays
- image + mask + ring avatar compositions
- background + content layering
- badge overlays

### 7.12 Fill distribution

For row/column layout:

- fixed and hug children consume their resolved base size first
- gaps are subtracted
- remaining main-axis size is split evenly across all fill children
- each fill child is then clamped through its own `Constraints`

### 7.13 Overflow scaling

Vertical parents may use `OverflowPolicy: scaleToFitY`.

If required content height exceeds available height:

- a scale factor is computed
- `layoutScale` is written to the parent
- `TransformSystem` multiplies it into world transform scale

This is a coarse "fit everything vertically" mode, not scrolling.

### 7.14 Important layout caveat

`Align` does not replace layout. It only affects placement within a stack or cross-axis logic in layout containers.

A node under `Layout: None` still fundamentally uses authored rect plus offset.

## 8. Transform and render pipeline

### 8.1 `ComputedRect`

This is the layout result:

- x
- y
- w
- h

It is local in the UI scene graph coordinate space.

### 8.2 `WorldTransform`

`TransformSystem` combines:

- local `TransformFx`
- `ComputedRect` position
- parent `WorldTransform`
- `Opacity`
- `LayoutScale`

The world transform stores:

- `x`
- `y`
- `scaleX`
- `scaleY`
- `rotation`
- `alpha`
- `visible`

### 8.3 Render backend abstraction

`UiRenderBackend` exposes operations like:

- create node by kind
- destroy node
- set parent
- set transform
- set rect
- set pivot
- set opacity/visibility
- set z-order
- set sprite
- set tile position
- set text
- set shape
- set shadow
- measure text
- measure sprite

Render node kinds are:

- `Frame`
- `Sprite`
- `NineSliceSprite`
- `TilingSprite`
- `Text`
- `Shape`

### 8.4 `RenderSyncSystem`

`RenderSyncSystem` is the bridge from ECS state to backend handles.

Main behavior:

- decide which entities need render handles
- destroy stale handles
- determine desired render kind from components
- create handle if needed
- attach to rendered parent handle
- push rect, sprite, text, shape, shadow, transform, opacity, and tiling updates

Handle kind resolution:

- sprite + nineSlice -> `NineSliceSprite`
- sprite + backgroundPan -> `TilingSprite`
- sprite only -> `Sprite`
- text -> `Text`
- shape -> `Shape`
- otherwise if it has children -> `Frame`

### 8.5 Text wrapping and style-driven measurement

Text is only re-sent to the backend when:

- the text string changed
- style id changed
- text style revision changed
- wrap width changed

If `Text.wrapWidthMode == 'NodeWidth'`, the render backend receives the node's computed width as wrap width.

This is how Royal Blush removed one-off width helper code and made wrapping data-driven.

## 9. Text measurement

`TextMeasureSystem` uses actual backend text handles.

It:

- queries text nodes
- builds a measure signature from text, style, and text style revision
- skips unchanged nodes
- asks the Pixi text object for `getLocalBounds()`
- writes `MeasuredSize`
- marks the world changed if size changed

This means hug-sized text uses real rendered bounds, not just string length estimation.

## 10. Image source resolution

`ImageSourceSystem` is a generic data-driven texture selector.

Modes:

- `Fixed`: choose first texture key
- `Random`: choose random texture key, optionally once and cached per entity
- `Hash`: stable hash selection from seed, scope, node id, and texture key set
- `Direct`: use the seed directly if it exists in textureKeys, else fallback to first

This makes profile avatars and list item portraits data-driven instead of hardcoded texture switches.

## 11. Text binding

`TextBindingSystem` maps runtime keys into authored text nodes.

A resolver function is supplied externally by the application layer.

Formatting options:

- `Plain`
- `Number`
- `Percent1`
- `Coins`
- `HandRank`

Result text is:

- `prefix + formattedValue + suffix`

This allows scenes to remain generic while app code supplies live stats and labels.

## 12. Masking

`MaskShapeSystem` supports two modes:

1. Use a real source node handle as the mask if `sourceNodeId` points to a renderable node.
2. If that is not available, create a fallback `Graphics` mask dynamically using the source bounds.

Mask shape kinds:

- `Circle`
- `RoundedRect`

A common pattern is:

- author a hidden or editor-locked mask source node
- reference it from an image node's `MaskShape.sourceNodeId`

That is exactly how the profile avatar uses a circular crop.

## 13. Safe area

`SafeAreaSystem` receives layout context from the outer app shell and writes normalized `SafeInsets` to all entities that have `SafeArea`.

`LayoutSystem` then includes those insets in effective padding.

This means authored nodes can opt in to safe-area compensation without every scene hardcoding iPhone notch offsets.

## 14. Runtime effects systems

These are not core layout, but they are part of the architecture.

### 14.1 Background pan

`BackgroundSystem`:

- advances parallax offsets over time
- supports tab-switch kick/parallax impulse
- stores offsets in `RuntimeOffset`

### 14.2 Wobble

`WobbleSystem`:

- applies sinusoidal offsets into `RuntimeOffset`
- supports one-wave and two-wave variants

### 14.3 Animation

`AnimationSystem`:

- triggers frame sequences on events like `OnTabSelected` or `OnAction`
- updates sprite texture keys over time
- supports looping and non-looping playback

### 14.4 Smoke

`SmokeSystem` exists as a specialized particle-ish effect runtime using `SmokeFx` authored data.

## 15. Data-backed virtualized lists

This is one of the most advanced RUICS features.

A virtualized list host is authored as a normal node with:

- `DataRequest`
- `ListView`
- often `MaskShape`
- often inside a `Stack` viewport

Example from ranking:

- `RankingListViewport` is the visible viewport and mask region
- `RankingListHost` is the scroll content host
- `RankingListHost` owns `DataRequest`, `ListView`, and `MaskShape`

### 15.1 DataRequest

`DataRequest` describes how to fetch records:

- `Top` mode means fetch first page by limit
- `AroundSelf` mode means fetch a window around the active account id

`buildDataRequestUrl(...)` builds query parameters for:

- initial load
- prepend page
- append page

### 15.2 Page extraction

`extractDataRequestPage(...)` reads items and cursors from response JSON.

The field paths are data-driven:

- `itemsField`
- `cursorBeforeField`
- `cursorAfterField`

### 15.3 ListView

`ListView` describes:

- which item scene to instantiate
- row height and gap
- optional cap for visible item count
- optional overscan rows
- bindings from item fields to node/component fields
- status scenes for loading, empty, and error states

### 15.4 Pooling and virtualization

Royal Blush does not instantiate one row entity per backend row.

Instead it:

- calculates how many rows should be alive based on viewport height
- optionally caps that count with `visibleItemCount`
- adds overscan rows if configured
- creates a reusable row pool of scene instances
- rebinds them as the scroll window changes

The important design rule is:

- pool size is stable while scrolling
- it only changes when viewport height changes

That prevents layout churn.

### 15.5 List item rebinding

Each pooled row is a scene instance of `itemSceneId`.

Bindings produce scene-instance params of the form:

```json
{
  "itemKey": "acct-123",
  "overrides": [
    {
      "nodeId": "RankingPlayerName",
      "component": "Text",
      "field": "text",
      "value": "Alice"
    }
  ]
}
```

Then `reapplySceneInstanceParams(...)` patches the row scene with the bound values.

### 15.6 Status scenes

Instead of special-case rendering in code, a list can switch to dedicated authored scenes for:

- loading
- empty
- error

That keeps list UX data-driven.

## 16. Authoring patterns

### 16.1 Reusable prefab style scenes

Small scenes such as:

- `profile-avatar`
- `ranking-profile-item`
- `chest-slot`

are used like prefabs.

They are mounted by `SceneInstance` and modified using params and overrides.

### 16.2 Overlay layers

A common pattern is:

- parent `Layout: Stack`
- child background node
- child content node
- child overlay node

This is used for:

- ranking jump buttons
- modal cards and backdrops
- highlight layers
- image + mask + border compositions

### 16.3 Dynamic action assignment

Authoring can use placeholder action ids in scene templates. Runtime params may then rewrite them.

Example:

- chest slot root has `actionId: chest.slot.0`
- a mounted scene instance can receive `slotIndex`
- runtime code mutates the root action id to `chest.slot.N`

### 16.4 Direct texture-driven avatars

The profile avatar scene uses `ImageSourceResolver` in `Direct` mode with a whitelist of allowed portrait texture keys.

That makes avatar selection deterministic and constrained by authored data.

## 17. Tooling and editor implications

RUICS was built with an editor in mind.

Important editor-facing features:

- stable `NodeId`
- `Name` for display labels
- registry-driven component discovery
- `EditorVisible` and `EditorLocked`
- round-trip serialization through `serializeLayoutV2` and `deserializeLayoutV2`
- auto-generated inspector component availability from the component registry

The Royal Blush inspector moved away from a stale hand-maintained list and now discovers components from the real registry. That is an important design lesson for Rae.

## 18. Important design lessons for reimplementation in Rae

If you rebuild this architecture in Rae, keep these design decisions:

1. Use stable node ids and a dictionary-of-nodes scene file format.
2. Keep layout components small and orthogonal.
3. Separate authored components from derived runtime components.
4. Make scene instancing first-class, not an afterthought.
5. Preserve root placement when mounting nested scenes.
6. Support generic per-node overrides through scene-instance params.
7. Make text wrapping data-driven through `wrapWidthMode`, not widget-specific code.
8. Make list rendering scene-driven and pooled.
9. Keep active/inactive nodes out of measurement and layout.
10. Let safe area be a component-level opt-in.
11. Keep ECS tables simple and debuggable.
12. Keep render backend abstract so the scene/runtime model is not tied to one renderer.

## 19. Suggested Rae implementation structure

If Rae is going to recreate this cleanly, a good module split would be:

- `ecs/entity`
- `ecs/componentTable`
- `ui/world`
- `ui/types`
- `ui/sceneFormat`
- `ui/sceneLoader`
- `ui/sceneInstance`
- `ui/layout`
- `ui/transform`
- `ui/renderBackend`
- `ui/renderSync`
- `ui/textMeasure`
- `ui/imageSource`
- `ui/textBinding`
- `ui/masking`
- `ui/listView`
- `ui/input`
- `ui/editorSupport`

And the scene file model should keep:

- one root id
- node dictionary
- explicit `Children` ordering
- component objects with no hidden inheritance

## 20. Short reference summary

RUICS is best understood as:

- an ECS-backed scene graph
- using JSON scene files with stable node ids
- with Flexbox-like but much simpler layout primitives
- with nested reusable scene instances
- with data binding and list virtualization built from authored components
- and with runtime systems that derive computed layout, transforms, and render handles deterministically

That combination is what makes it suitable for recreation in a language like Rae.
